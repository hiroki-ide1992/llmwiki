# リバースプロキシ

クライアント（ブラウザ）とバックエンドサーバーの間に立ち、クライアントからのリクエストを代理で転送するサーバー。

## 基本構造

```
クライアント → リバースプロキシ → バックエンドサーバー
```

クライアントから見えるアクセス先はリバースプロキシのみ。バックエンドサーバーのアドレスは隠蔽される。

## フォワードプロキシとの違い

| | フォワードプロキシ | リバースプロキシ |
|---|---|---|
| 代理対象 | クライアント側 | サーバー側 |
| 設置者 | クライアント/ユーザー | サービス提供者 |
| 典型用途 | 匿名アクセス・フィルタリング | 負荷分散・SSL終端・CORS回避 |

## 主な用途

- **[[CORS]] 回避** — ブラウザが直接他オリジンへアクセスしないように中継する
- **[[負荷分散]]** — 複数のバックエンドサーバーへリクエストを振り分ける
- **SSL終端** — HTTPS→HTTP変換をプロキシ側で処理する
- **キャッシュ** — 静的コンテンツをキャッシュしてバックエンドの負荷を軽減
- **セキュリティ** — バックエンドサーバーのIPアドレスを隠蔽する

## CORS との関係

ブラウザはリバースプロキシを「同一オリジン（または許可されたオリジン）のサーバー」として認識するため、[[CORS]] 制限が発動しない。裏側でリバースプロキシが外部サーバーへアクセスするが、この通信はサーバー間通信なので CORS の対象外。

→ 詳細: [[CORS#リバースプロキシによる回避]]

## 代表的な実装

- [[nginx]] — 最も広く使われるリバースプロキシ兼Webサーバー
- Apache (mod_proxy)
- HAProxy
- Caddy
- Cloudflare（CDN兼リバースプロキシ）
