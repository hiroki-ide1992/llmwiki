# CORS (Cross-Origin Resource Sharing)

ブラウザに組み込まれたセキュリティ機構。あるオリジンで読み込まれたページが、異なるオリジンのリソースへ JavaScript からアクセスすることを制限する。

## オリジンの定義

`プロトコル + ホスト + ポート` の組み合わせ。一つでも違えば「別オリジン」。

例:
- `http://localhost:3000` と `https://example.com` → 別オリジン
- `http://localhost:3000` と `http://localhost:8080` → 別オリジン（ポートが違う）

## CORSが守っているもの

悪意あるWebページが、ログイン済みのユーザーのブラウザを経由して、別サイト（例: 銀行）の情報を無断で読み取ることを防ぐ。

```
悪意あるサイト → fetch("https://bank.example.com/account")
                ↑
           ブラウザがブロック（銀行サイトが許可していないため）
```

## 許可の仕組み

サーバー側が以下のHTTPレスポンスヘッダーで許可オリジンを明示する:

```http
Access-Control-Allow-Origin: https://trusted-site.com
```

`*` を指定すると全オリジンからのアクセスを許可（認証情報なしのリクエストのみ）。

## CORS の適用範囲

**対象**: ブラウザ上の JavaScript からの HTTP リクエスト
**対象外**: サーバー側のプログラム（nginx・Node.js・Python・curl など）

## リバースプロキシによる回避

[[リバースプロキシ]]を使うと、ブラウザが直接外部サイトへアクセスしなくなるため CORS が発動しない。

```
（CORSあり）  ブラウザ → 外部サイト  ← オリジン違反でブロック
（回避後）    ブラウザ → nginx → 外部サイト
                  ↑ ブラウザは localhost にアクセスしているだけ
```

> 正確には CORS を「突破」しているのではなく、CORS 判定の対象を nginx（同一オリジン）に変えているだけ。

## スクレイピングとの注意点

CORS を回避できることと、スクレイピングが許可されることは別問題。アクセス先の利用規約・robots.txt・アクセス頻度・著作権を必ず確認すること。
