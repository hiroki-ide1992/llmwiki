# リバースプロキシとは

たとえばこうです。

ブラウザ
  ↓
http://localhost:8080
  ↓ nginx
  ↓
https://example.com

ブラウザから見ると、アクセス先はあくまで localhost:8080 です。
でも実際には、nginx が裏側で example.com に取りに行って、その結果をブラウザに返しています。

なぜ CORS 制限に引っかからないのか

CORS は正確には ブラウザが課す制限です。

重要なのはここです。

ブラウザ → 他サイト

この通信では、ブラウザが

「今開いているページのオリジンと、アクセス先のオリジンが違うな」
「相手サーバーが許可していないならブロックしよう」

と判断します。

たとえば、ローカルの画面から直接こうアクセスするとします。

fetch("https://example.com/api")

ブラウザから見ると、

http://localhost:3000 → https://example.com

なので オリジンが違う。
そのため、example.com 側が

Access-Control-Allow-Origin: http://localhost:3000

のように許可していないと、ブラウザがレスポンスを JavaScript に渡しません。

nginx を挟むと何が変わるか

nginx をこう使ったとします。

ブラウザ → http://localhost:8080/proxy/example
nginx   → https://example.com

この場合、ブラウザから見る通信は

http://localhost:3000 → http://localhost:8080

または、同じポートにまとめていれば

http://localhost:3000 → http://localhost:3000

のようになります。

つまり、ブラウザは「localhost にアクセスしているだけ」と認識するわけです。

その裏で nginx が https://example.com にアクセスしますが、

nginx → example.com

これは サーバー間通信です。

CORS はブラウザの制限なので、nginx や curl や Python や Node.js のようなサーバー側通信には基本的に適用されません。

つまり原理はこれ

直接アクセス：

ブラウザ → 他サイト
        ↑
      CORSで制限される

リバースプロキシ経由：

ブラウザ → nginx → 他サイト
   ↑
ブラウザから見ると nginx にアクセスしているだけ

なので、ブラウザの CORS 制限を回避できたように見えます。

正確には、CORSを突破しているのではなく、ブラウザがCORS判定する対象を nginx に変えているということです。

nginx のイメージ

かなり単純化すると、nginx 側ではこういうことをしています。

location /proxy/ {
    proxy_pass https://example.com/;
}

ブラウザが

http://localhost:8080/proxy/page

にアクセスすると、nginx が裏で

https://example.com/page

へ取りに行きます。

そして取得した HTML や JSON を、nginx がブラウザに返します。

CORS とスクレイピングは別問題

CORS はあくまで ブラウザ上の JavaScript から他オリジンへアクセスする制限です。

なので、以下のようなアクセスは CORS の対象外です。

nginx
Node.js
Python
curl
サーバーサイドのバッチ処理

そのため、スクレイピング用途では普通はブラウザから直接取るよりも、サーバー側で取得します。

ただし、CORSを回避できることと、スクレイピングしてよいことは別です。

相手サイトの利用規約、robots.txt、アクセス頻度、著作権、ログイン後ページかどうかなどには注意が必要です。特にログインが必要なページや有料コンテンツ、過度なアクセスは問題になりやすいです。

もう少し技術的に言うと

CORS で守られているのは、主にこれです。

「悪意あるWebページが、ユーザーのブラウザを使って、
別サイトの情報を勝手に読めないようにする」

例えばユーザーが銀行サイトにログイン済みの状態で、悪意あるサイトを開いたとします。

その悪意あるサイトの JavaScript が銀行サイトへ勝手に fetch() して内容を読むことができたら危険です。

そこでブラウザは、

このページの出どころと、アクセス先が違う
↓
相手が明示的に許可していなければ JS には渡さない

という制限をします。

でも nginx はブラウザではありません。
なので nginx 自体は CORS に縛られません。

まとめ

リバースプロキシで CORS を回避できた理由は、

ブラウザが直接他サイトへアクセスしていないからです。

実際の構造はこうです。

ブラウザ → 自分のnginx
nginx → 他サイト

ブラウザから見るとアクセス先は自分の nginx なので、CORS 的には問題になりにくい。
他サイトへのアクセスは nginx がサーバー側で行うため、ブラウザの CORS 制限を受けません。
