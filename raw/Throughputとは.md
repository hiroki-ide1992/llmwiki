# Throughput（スループット） とは、ざっくり言うと

一定時間あたりに処理できる量

のことです。

たとえば、道路でいうと「1分間に何台の車が通れるか」に近いです。

例

Web/API なら：

観点	Throughput の例
リクエスト処理量	1秒あたり何リクエスト処理できるか
データ転送量	1秒あたり何MB送れるか
DB処理量	1秒あたり何件のクエリを処理できるか
メッセージ処理量	1秒あたり何件のメッセージを処理できるか

よく見る単位はこんな感じです。

RPS / req/s：1秒あたりのリクエスト数
TPS：1秒あたりのトランザクション数
MB/s / GB/s：1秒あたりのデータ転送量
IOPS：1秒あたりのI/O回数
レイテンシとの違い

ここが大事です。

用語	意味
Throughput	どれだけ大量に処理できるか
Latency	1件の処理にどれだけ時間がかかるか

例えると：

Throughput
→ レジが1時間に何人さばけるか

Latency
→ 1人のお客さんが会計完了するまで何秒かかるか

インフラで問題になる例

たとえば API サーバーで、

CPU が高い
DB接続数が足りない
ネットワーク帯域が足りない
OpenSearch の search queue が詰まる
ECSタスク数が少ない
ALB配下のサーバーが処理しきれない

みたいな状態になると、Throughput が頭打ちになります。

つまり、アクセスが増えても処理量が増えず、待ち行列ができて、結果的にレイテンシも悪化します。

一言でいうと

Throughput = システムがどれだけの量をさばけるかを表す性能指標

です。
インフラ監視では「CPU使用率」や「レイテンシ」と一緒に見て、どこがボトルネックで処理量が伸びないのかを調べるときに使います。
