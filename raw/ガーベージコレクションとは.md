インフラにおける**ガーベージコレクション（Garbage Collection / GC）**とは、主に メモリ上の不要になったデータを自動で掃除する仕組み のことです。

特にインフラ監視では、Java系アプリケーションや OpenSearch / Elasticsearch / Solr などの JVM を使うミドルウェアでよく出てきます。

ざっくり言うと

アプリや検索エンジンは、処理中にメモリを使います。

たとえば OpenSearch なら、

検索リクエストの処理
クエリ結果の一時データ
キャッシュ
インデックス処理
集計処理

などでメモリを使います。

処理が終わって不要になったメモリを、JVM が自動で回収するのが ガーベージコレクション です。

例えると

作業机で考えるとわかりやすいです。

作業中に資料やメモを机に広げる
→ 作業が終わった紙は不要になる
→ 机がいっぱいになる前に片付ける

この「不要な紙を片付ける作業」が GC です。

インフラで問題になる理由

GC 自体は必要な処理ですが、問題は GC 中にアプリの処理が一時的に止まったり遅くなったりすることがある 点です。

これを Stop The World と呼ぶことがあります。

つまり、

GCが走る
↓
メモリ掃除のために一部処理が止まる
↓
リクエスト処理が遅くなる
↓
レスポンス遅延、タイムアウト、5xx につながることがある

という流れです。

OpenSearch で見ると

OpenSearch では JVM メモリ、つまり JVM Heap を使います。

この Heap が高くなりすぎると、GC が頻繁に発生します。

特に見るポイントはこのあたりです。

見るもの	意味
JVM Memory Pressure	JVMメモリの使用率
GC回数	GCがどれくらい頻繁に起きているか
GC時間	GCにどれくらい時間がかかっているか
CPU使用率	GCでCPUを使いすぎていないか
Search latency	検索遅延が出ていないか
5xx / timeout	障害につながっていないか
良くない状態の例
JVM Memory Pressure が 80% 超え
GC が頻繁に発生
GC時間が長い
検索 latency が上昇
search queue が溜まる
5xx が増える

この場合、メモリ不足や重いクエリ、シャード設計、キャッシュ肥大などを疑います。

よくある原因

GC が重くなる原因は、たとえば以下です。

JVM Heap が足りない
重い検索クエリが多い
aggregation が重い
fielddata / cache を使いすぎている
shard 数が多すぎる
index サイズに対してノードが少ない
bulk indexing が多い
メモリリークに近い状態になっている
重要な考え方

GC は「悪いもの」ではありません。

ただし、インフラ運用では、

GCが適度に動いている → 正常
GCが頻繁すぎる / 長すぎる → 性能劣化や障害の兆候

と見ます。

一言でいうと

ガーベージコレクションとは、JVMなどが不要になったメモリを自動で片付ける仕組み。
ただし、頻繁に起きたり時間が長いと、検索遅延・CPU高騰・5xxの原因になる。

OpenSearch や Solr で言えば、JVM Memory Pressure が高いときに一緒に確認すべき重要な指標です。
